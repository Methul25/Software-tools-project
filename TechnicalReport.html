1. Introduction & Development Environment

The BlindMatch Project Approval System (PAS) is a secure, web-based platform built with ASP.NET Core 8 MVC that facilitates the allocation of final-year research project supervisors to students. It prioritises a "blind matching" mechanism in which supervisors review anonymous project proposals, eliminating identity-based unconscious bias during the initial selection phase. The system serves four distinct roles: Student, Supervisor, Module Leader, and System Administrator, each isolated via ASP.NET Core Role-Based Access Control (RBAC).

The development environment consists of the .NET 8 SDK, Visual Studio Code, and the Entity Framework Core CLI. Persistence is handled by SQLite in development (configured in appsettings.Development.json via the connection string DefaultConnection) with a deliberate comment in Program.cs indicating the swap point to UseSqlServer() for production deployment. The application is structured following the MVC separation of concerns pattern, with a dedicated service layer (IMatchingService / MatchingService) injected via the ASP.NET Core DI container as a scoped service. Password policy is enforced at the Identity layer: minimum eight characters, requiring at least one digit, one uppercase, one lowercase, and one non-alphanumeric character.

2. Blind-Match Logic and State Management

2.1 The Anonymity Problem

In traditional supervisor allocation workflows, supervisors receive a list of proposals alongside the student's name and student ID. Research in academic allocation systems suggests this introduces affinity bias — supervisors unconsciously favour students they recognise (Hinton, 2015). The Blind-Match mechanism addresses this by separating the phases of selection and reveal: a supervisor may only read a proposal's Title, Abstract, Technical Stack, and Research Area until they explicitly confirm a match.

2.2 State Machine

Every proposal moves through a formally defined state machine encoded in the ProposalStatus enum (Models/ProposalStatus.cs):

Pending → UnderReview → Matched
UnderReview → Pending  (if supervisor withdraws)
Pending / UnderReview → Withdrawn  (student withdraws)

The five enum values are Pending = 0, UnderReview = 1, Matched = 2, Withdrawn = 3, and Reassigned = 4. Transitions are exclusively managed through MatchingService, meaning no controller directly mutates the proposal status. This architectural decision means the business rules are testable in isolation without an HTTP layer, directly supporting the unit testing strategy described in Section 4.

2.3 The Identity-Reveal Mechanism

The reveal mechanism centres on a single Boolean column: IsRevealed on the ProjectMatch entity. It defaults to false at match creation (ExpressInterestAsync) and is irreversibly set to true only inside ConfirmMatchAsync:

// MatchingService.cs — ConfirmMatchAsync
match.ConfirmedAt = DateTime.UtcNow;
match.IsRevealed = true;
match.SupervisorNote = note;

match.Proposal.Status = ProposalStatus.Matched;
match.Proposal.UpdatedAt = DateTime.UtcNow;

await _context.SaveChangesAsync();

Both the supervisor dashboard and the student dashboard independently query the IsRevealed flag from the database before rendering identity information. In StudentController.Dashboard, the projection explicitly guards the reveal:

// StudentController.cs — BuildStatusViewModel
SupervisorName = p.Match?.IsRevealed == true ? p.Match.Supervisor?.FullName : null,
SupervisorEmail = p.Match?.IsRevealed == true ? p.Match.Supervisor?.Email : null

This means that even if a front-end rendering bug occurred, the controller layer would never pass supervisor identity to the view unless the database flag is set. The anonymity guarantee is therefore enforced at two independent layers: service and controller.

2.4 Anonymity at the Service Layer

Crucially, the LINQ query inside GetProposalsForSupervisorAsync does not call .Include(p => p.Student). This is a deliberate design choice: Entity Framework Core will only populate the Student navigation property if it is explicitly included. The integration tests confirm this invariant by querying through a fresh DbContext instance (to bypass the EF change-tracker identity cache) and asserting that results[0].Student is null:

// Integration/DatabaseIntegrationTests.cs
await using var freshContext = TestDbContextFactory.Create(_dbName);
var freshService = new MatchingService(freshContext);
var proposals = (await freshService.GetProposalsForSupervisorAsync(_supervisor.Id)).ToList();
Assert.Null(proposals[0].Student); // anonymity enforced

2.5 Reassignment

Module Leaders (and System Administrators) may override the automatic matching via ReassignProposalAsync. This method removes the existing match record, creates a new unconfirmed match for the designated supervisor, and resets the proposal to UnderReview. The new supervisor must still explicitly confirm the match, meaning identity reveal does not happen implicitly during reassignment. This preserves the integrity of the blind-match guarantee even in coordinator-driven workflows.

3. Version Control Strategy

3.1 Conventional Commits

The project adopted the Conventional Commits 1.0.0 specification for all commit messages. Each message follows the format <type>(<scope>): <description>. Types used are: feat (new feature), fix (bug correction), test (test additions), chore (tooling/config), and refactor (non-functional restructuring). The scope component identifies the architectural layer affected: models, db, service, controller, ui, and test.

This convention was chosen because it: (1) makes the commit log machine-readable and compatible with automated changelog generation tools such as conventional-changelog; (2) forces contributors to reason about the type of change they are making before committing, reducing mixed-concern commits; and (3) produces a log that directly maps to the feature delivery timeline reviewers can follow.

3.2 Sprint-Driven Commit Flow

Development was organised into six sprints, each with a clear deliverable and a corresponding cluster of commits. The commit log (GIT_COMMIT_HISTORY.sh) demonstrates the following progression:

This granular, sprint-gated approach ensures that every logical layer is independently verifiable in version history. If a regression is introduced at the service layer, git bisect can identify the exact commit responsible by replaying history sprint by sprint.

3.3 Dependency Injection and Clean Architecture

The architectural decision to register IMatchingService as a scoped service in Program.cs (builder.Services.AddScoped<IMatchingService, MatchingService>()) directly supports version control cleanliness: because all matching logic lives in one service class rather than scattered across controllers, future changes to the business rules require editing a single file. This minimises diff noise across branches and reduces the risk of merge conflicts on controller files.

4. Testing Strategy and Critical Analysis

4.1 Overview and Tool Selection

The testing suite comprises 90 tests across four distinct layers: unit tests (47 across two test classes), integration tests (22 across two test classes), mock/controller tests (14), and functional end-to-end tests (7). xUnit.net 2.6 was chosen as the test framework, Moq 4.20 for mock creation, and the EF Core InMemory provider for database-level tests. The decision to use xUnit over alternatives such as NUnit or MSTest was motivated by three factors: xUnit's IAsyncLifetime interface allows per-test async setup and teardown without additional attributes; its constructor injection model makes test dependencies explicit; and it is the native framework used internally by the ASP.NET Core team, ensuring maximum compatibility with Microsoft.AspNetCore.Mvc.Testing.

4.2 Unit Tests — Matching Logic and Controller Behaviour

The unit layer contains two test classes. Unit/MatchingServiceTests.cs (33 tests) exercises all seven methods of MatchingService against an in-memory store, covering every state transition in the proposal lifecycle. Unit/StudentControllerTests.cs (14 tests) tests the StudentController in isolation, asserting correct redirect targets, TempData messages, and model state handling for all five actions: Dashboard, Submit (GET + POST), Edit (GET + POST), and Withdraw.

Each test class receives a uniquely named in-memory database via Guid.NewGuid().ToString(), ensuring complete isolation between test runs regardless of execution order. Parameterised tests (xUnit's [Theory] / [InlineData]) are used where the same logic applies to multiple input states, such as verifying that both Matched and Withdrawn proposals are unavailable for matching:

// Unit/MatchingServiceTests.cs
[Theory]
[InlineData(ProposalStatus.Matched)]
[InlineData(ProposalStatus.Withdrawn)]
public async Task IsProposalAvailable_WhenFinalStatus_ReturnsFalse(ProposalStatus status)
{
    var proposal = await TestDbContextFactory.SeedProposalAsync(_context, _student.Id, _area.Id, status);
    var result = await _service.IsProposalAvailableForMatchingAsync(proposal.Id);
    Assert.False(result);
}

This approach reduces duplication while exhaustively testing the state machine boundaries discussed in Section 2.2. A key debate during test design was whether to assert on returned MatchResult objects or to query the database directly. The chosen strategy does both: the return value is asserted for the caller's perspective, and the database state is independently verified to ensure the persistence layer reflects the expected change. This double-assertion approach detects bugs where a method returns a success message but fails to commit the change.

4.3 Integration Tests — Database Interactions

The integration layer contains two test classes. Integration/DatabaseIntegrationTests.cs (11 tests) verifies multi-step workflows: the flagship test FullWorkflow_Submit_Express_Confirm_PersistsCorrectState exercises the full lifecycle — seed → express interest → confirm — asserting intermediate state after each step. Integration/ProposalRepositoryTests.cs (11 tests) focuses on the persistence layer in isolation, covering proposal CRUD, status-transition queries, research-area filtering by expertise, cascade delete behaviour, and anonymity enforcement. Together, these files ensure that the storage contracts assumed by the service layer are honoured by EF Core.

A notable technical challenge arose with the anonymity invariant test. The EF Core InMemory provider maintains an internal identity map (change tracker) that automatically resolves navigation properties from already-tracked entities, regardless of whether .Include() was called. This mirrors the behaviour of lazy loading rather than the explicit loading required by the production SQL Server backend. The solution was to query through a second, fresh DbContext instance sharing the same named in-memory database, which has an empty change tracker:

await using var freshContext = TestDbContextFactory.Create(_dbName);
var freshService = new MatchingService(freshContext);
var proposals = (await freshService.GetProposalsForSupervisorAsync(_supervisor.Id)).ToList();
Assert.Null(proposals[0].Student);

While the EF InMemory provider is widely used for integration testing, it has a documented limitation: it does not enforce relational constraints such as foreign keys or unique indexes. To mitigate this, the unique index test (SupervisorExpertise_UniqueConstraint_PreventsDoubleEntry) validates the constraint declaration via EF Core's metadata API (_context.Model.FindEntityType()) rather than relying on an exception, ensuring that the constraint will be honoured when deployed against a real relational engine.

4.4 Mock Tests — Isolated Controller Contracts

The Moq-based tests in Mocks/SupervisorControllerMockTests.cs (14 tests) decouple the controller from its service dependency, enabling precise assertions about the controller's responsibilities: correct delegation to the service, correct redirect targets, and correct TempData population. The MockUserManagerHelper wraps the twelve-parameter UserManager constructor into a reusable factory:

// Helpers/MockUserManagerHelper.cs
var store = new Mock<IUserStore<ApplicationUser>>();
return new Mock<UserManager<ApplicationUser>>(
    store.Object, optionsAccessor.Object, passwordHasher.Object,
    userValidators, passwordValidators, keyNormalizer.Object,
    errors.Object, services.Object, logger.Object);

A deliberate test asserts that confirming a match never triggers an ExpressInterestAsync call (ConfirmMatch_NeverCallsExpressInterest), guarding against accidental re-expression of interest during a confirmation flow. This type of negative-behaviour assertion is a testing pattern that pure state-based assertions cannot cover.

4.5 Functional Tests — End-to-End User Journeys

The functional test class Functional/BlindMatchUserJourneyTests.cs (7 tests) simulates complete user journeys through the service layer and database, without a running HTTP server. Each test exercises a realistic scenario from a specific role's perspective:

JourneyScenario
1Full happy path: student submits → supervisor browses anonymously → expresses interest → confirms → identity revealed
2Two supervisors compete: only the first to express interest succeeds; the proposal blocks the second
3Withdrawal blocked: a student cannot withdraw a proposal once it is UnderReview (supervisor match exists)
4Supervisor withdraws: proposal returns to Pending, match record deleted, a different supervisor can now claim it
5Admin reassign: new match is unconfirmed, requires the new supervisor to explicitly confirm before reveal
6Anonymity guard: before confirmation, querying the match through a fresh DbContext returns null for the student navigation property
7Module Leader oversight: confirmed matches are queryable with both participant identities loaded

A significant debugging challenge arose in Journey 6. The shared _context instance had the student entity tracked in its change tracker from the InitializeAsync seed phase. EF Core's identity map supplied the Student navigation property even without an explicit .Include(), causing an Assert.Null failure. The fix — querying the match through the same blindContext used for the service call (a separate context instance sharing the in-memory DB name) — corrects the test's semantics and accurately replicates real SQL Server behaviour, where navigation properties are strictly opt-in via .Include().

5.2 Reflection on Technical Challenges

Challenge 1: EF Core cascade delete cycles. The initial ApplicationDbContext configuration triggered a SQL Server migration error because multiple delete paths existed from ApplicationUser to ProjectProposal (as student author and as supervisor via match). This was resolved by explicitly setting DeleteBehavior.Restrict on the ProjectProposal → Student and ProjectMatch → Supervisor relationships in OnModelCreating, leaving only the one-to-one ProjectMatch → Proposal as Cascade. This was a meaningful constraint-modelling decision: a supervisor's account deletion should not silently cascade across student proposals they never authored.

Challenge 2: UserManager mock complexity. ASP.NET Core's UserManager<TUser> has a constructor with twelve parameters, all required to be non-null or Moq would throw ArgumentNullException during setup. The solution was the MockUserManagerHelper static factory class, which centralises the twelve-mock boilerplate and exports a pre-configured Mock<UserManager<ApplicationUser>>. This also reduced the per-test setup code from approximately 25 lines to a single call, demonstrating the DRY principle in the test infrastructure itself.

Challenge 3: EF InMemory identity cache breaking anonymity tests. As described in Section 4.3, the EF Core InMemory provider resolves navigation properties from its in-process identity map even when they are not explicitly included in a query. This caused two tests asserting proposal.Student == null to fail, because the student entity had been tracked by the same context during the seed phase. The fix — using a shared database name but a new context instance — corrects the test's semantics to accurately represent the behaviour of the production SQL Server backend, where navigation properties are only populated through explicit Include() calls.

Challenge 4: Regex validation scope. The coursework rubric explicitly requires [RegularExpression] data annotations for data validation. Initial ViewModels used only [Required], [StringLength], and [EmailAddress]. To satisfy this requirement, [RegularExpression] was added to three fields: InstitutionId (enforces the institutional ID format ^[A-Za-z]{1,3}\d{4,10}$, e.g. ST20240001), Department (alphanumeric with common punctuation), and TechnicalStack (must start with a letter and may contain technology-common symbols such as #, +, and /). The regex patterns were chosen to be permissive enough for legitimate academic data while blocking values that would indicate erroneous input, such as purely numeric institution IDs or technical stacks beginning with special characters.

Challenge 5: ProposalStatus.Reassigned is unused. The domain analysis identified that the Reassigned = 4 enum value, while correctly declared, is never assigned by ReassignProposalAsync; the method instead sets the status to UnderReview. This is a deliberate design decision: since a reassigned proposal still requires the new supervisor to review and confirm it, the workflow state is semantically identical to UnderReview. The Reassigned value is retained in the enum for future differentiation, such as displaying a distinct "Reassigned by Coordinator" badge in the student dashboard.

6. Conclusion

The BlindMatch PAS successfully implements a secure, bias-mitigating proposal allocation workflow across four user roles in ASP.NET Core 8 MVC. The blind-match state machine, managed atomically through a single service layer and governed by the IsRevealed flag on ProjectMatch, provides a robust and independently testable guarantee that student identity is never exposed to supervisors prior to confirmation. The finalised testing suite of 90 tests, all passing — spanning unit (47), integration (22), mock (14), and functional (7) layers — exceeds 80% coverage of the matching logic and validates both positive-behaviour and negative-behaviour invariants. The [RegularExpression] data annotations, Conventional Commits strategy, and sprint-gated commit structure together provide a codebase that is professionally structured, defensively validated, and fully auditable. The technical challenges encountered — EF Core cascade cycles, UserManager mock complexity, in-memory provider identity-cache semantics, and regex validation scope — each produced reusable architectural patterns that improve the maintainability of the codebase beyond the immediate coursework context.
