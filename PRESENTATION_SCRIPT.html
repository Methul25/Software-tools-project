Good morning / good afternoon. My name is [NAME], and on behalf of our team, welcome to the presentation of BlindMatchPAS — our submission for PUSL2020: Software Development Tools and Practices.

The question we started with was this: in a university setting, when supervisors see a list of project proposals alongside student names and IDs, do they choose the proposal on merit, or on familiarity? Research by Hinton (2015) shows that unconscious affinity bias is real and measurable. Our system removes that variable entirely.

BlindMatchPAS is a fully functional web application that separates the act of selecting a proposal from the act of knowing who wrote it. A supervisor reviews, shortlists, and confirms a match before the student's identity is ever shown. The reveal only happens at the moment of mutual confirmation — by design, at the database level, not just the UI level.

Today we will walk you through the system architecture, the matching logic, a live demonstration, our version control strategy, and a comprehensive testing suite comprising 90 tests across four distinct layers. Each of my colleagues owns a specific area, and together we will cover every aspect of our technical report.

[SLIDE or SCREEN: Show the home page and shared layout in the browser or editor]

My own contribution to the codebase is the UI shell — the parts of the application that every user sees regardless of their role. Specifically, I built the shared _Layout.cshtml, which provides the navigation bar, role-sensitive menu links, and the consistent page wrapper used throughout the app. I also built the separate _AuthLayout.cshtml used on login and registration pages, where a minimal, centred layout is more appropriate than the main nav.

Alongside the layout, I authored the two ASP.NET Core Identity account pages: Login.cshtml and Register.cshtml, each backed by a PageModel class. These scaffold into the Identity middleware pipeline — so authentication, password hashing, and session cookie management are handled by the framework, and the pages I wrote are purely the UI and input-validation layer on top. The _LoginPartial.cshtml and _ValidationScriptsPartial.cshtml components are also mine — they inject the login/logout link into the nav bar and ensure client-side validation is wired up consistently across all forms.

[SLIDE: Project structure diagram or tech-stack list — ASP.NET Core 8 / EF Core / SQLite / Identity / xUnit]

Thank you, [NAME 1]. BlindMatchPAS is built on ASP.NET Core 8 MVC. We chose this framework because it enforces separation of concerns natively, has built-in Identity for role-based access control, and the dependency injection container is first-class — which had a direct positive impact on our testing strategy.

For persistence, we used Entity Framework Core with SQLite in development. That means the application runs entirely self-contained: clone the repository, run dotnet run, and the database is created and seeded automatically. The connection string and a single-line swap to UseSqlServer() is all that separates the development configuration from a production deployment.

The four user roles — Student, Supervisor, Module Leader, and System Administrator — are enforced at every controller action via [Authorize(Roles = "...")]. This is not a UI-level guard; it is enforced by the ASP.NET Core middleware pipeline, so route manipulation cannot bypass it.

The most important architectural decision was the service layer. All business logic lives in one class: MatchingService, implementing the IMatchingService interface. It is registered in Program.cs as a scoped service and injected into every controller that needs it. Controllers never touch the database directly. This single decision is what made writing 90 isolated, fast tests straightforward.

[SLIDE: State machine diagram — Pending → UnderReview → Matched, with withdrawal and reassignment arrows]

Thank you, [NAME 2]. The heart of the system is the ProposalStatus state machine, encoded as a C# enum with five values: Pending, UnderReview, Matched, Withdrawn, and Reassigned.

Every proposal starts as Pending when a student submits it. When a supervisor expresses interest, the proposal moves to UnderReview and a ProjectMatch record is created with IsRevealed = false. When the supervisor confirms, it transitions to Matched and, crucially, IsRevealed is set to true. That Boolean is the entire identity-reveal gate.

The anonymity guarantee is enforced at two independent layers. First, the service layer's GetProposalsForSupervisorAsync deliberately omits .Include(p => p.Student) in its LINQ query — EF Core will not populate the student navigation property unless explicitly asked. Second, in StudentController, the supervisor's name is only projected into the ViewModel if IsRevealed == true in the database. Even if a front-end rendering bug occurred, the controller would never pass supervisor identity to the view prematurely.

Reassignment, which any Module Leader or Admin can trigger, removes the existing match, creates a new unconfirmed match for the new supervisor, and resets the proposal to UnderReview. The new supervisor must still confirm, so the blind-match guarantee is preserved even in coordinator-driven workflows.

Thank you, [NAME 4]. Every commit in this repository follows the Conventional Commits 1.0.0 specification. The format is: type(scope): description. For example, feat(service): implement IMatchingService immediately tells any reviewer that this commit adds a feature to the service layer. The types we used are: feat, fix, test, chore, and docs.

This was not cosmetic. Conventional Commits forces each contributor to classify the type of change they are making before committing. It prevents mixed-concern commits — for example, bundling a model change with a test change in one commit, which is notoriously difficult to roll back cleanly. Every architectural layer is individually addressable in history.

Our ten commits map to six development sprints. Sprint 1 initialised the project. Sprint 2 built the domain model and migration. Sprint 3 delivered the service layer. Sprint 4 added controllers. Sprint 5 covered UI. Sprint 6 added all test classes, one commit per test file — unit, integration, and functional each in their own commit.

If a regression appeared in the service layer, git bisect could binary-search the commit history and pinpoint the exact change responsible in under a minute. That is the practical value of granular, well-scoped commits beyond just a clean-looking log.

Thank you, [NAME 5]. Our test suite runs 90 tests, zero failures. They are organised into four layers: 47 unit tests, 22 integration tests, 14 mock tests, and 7 functional end-to-end tests.

We chose xUnit.net over NUnit or MSTest for three specific reasons. First, xUnit's IAsyncLifetime interface provides per-test async setup and teardown without extra attributes — essential given our database-seeding requirements. Second, its constructor injection model makes dependencies explicit. Third, it is the framework used internally by the ASP.NET Core team, which guarantees compatibility with the ecosystem.

The unit layer has two classes. MatchingServiceTests (33 tests) exercises every method of MatchingService in isolation — every state transition, every failure path, every edge case. StudentControllerTests (14 tests) verifies the controller independently: correct redirects, correct TempData messages, and ownership guards that prevent a student from editing another student's proposal even by manipulating the route ID.

We used xUnit's [Theory] and [InlineData] for parameterised tests. For example, a single test body verifies that both Matched and Withdrawn proposals are unavailable for matching — without duplicating the test code. Our assertion strategy is deliberate: we assert both the return value and the database state independently, so a method that returns success but forgets to call SaveChangesAsync will be caught.

The mock layer has 14 tests in SupervisorControllerMockTests. These use Moq to substitute the real service with a programmable mock, which lets us assert things that state-based tests cannot: for example, that confirming a match never calls ExpressInterestAsync. That is a negative-behaviour assertion — it guards against accidental side effects, not just correct outcomes. We also built a MockUserManagerHelper to wrap ASP.NET Core's twelve-parameter UserManager constructor, reducing per-test boilerplate from 25 lines to one.

The functional layer has 7 journey tests in BlindMatchUserJourneyTests. These simulate full end-to-end scenarios through the service layer: the happy path from submission to reveal, two supervisors competing for the same proposal, student withdrawal blocks, supervisor withdrawal and re-availability, admin reassignment, the anonymity guard, and Module Leader oversight. One of these journeys uncovered a real bug: the EF Core identity cache was supplying a navigation property from the change tracker rather than the query — breaking the anonymity assertion. Fixing it required querying through a fresh context. [NAME 8] will explain that challenge alongside the others.

Thank you, [NAME 7]. Let me put the test suite in perspective.

Across four layers — 47 unit, 22 integration, 14 mock, 7 functional — the suite covers all core business rules, every state transition, negative-behaviour contracts, and end-to-end user journeys. Every test runs against an isolated in-memory database seeded fresh in InitializeAsync. The weakness we acknowledge openly is that the EF Core InMemory provider does not enforce relational constraints the way SQL Server does. We mitigate this by validating unique-index declarations through EF Core's metadata API rather than relying on an exception that the in-memory provider would never throw.

We encountered five notable technical challenges. The first was EF Core cascade-delete cycles — resolved by setting DeleteBehavior.Restrict on specific relationships. The second was mocking the twelve-parameter UserManager — resolved by the reusable MockUserManagerHelper factory. The third was the EF identity-cache bug in the anonymity tests, resolved by querying through a fresh context. The fourth was ensuring [RegularExpression] annotations were present on all relevant fields as required by the rubric. The fifth was a deliberate design decision to retain the Reassigned enum value for future use, even though the current implementation maps reassignment to UnderReview semantically.

In conclusion: BlindMatchPAS is a professionally structured, defensively validated system. The blind-match guarantee — enforced at service, controller, and database levels simultaneously — is not an interface detail but a verifiable architectural property.
